# Prompting playbook

Готовые к использованию шаблоны промптов для First-Agent и принципы, по которым
мы их пишем. Все шаблоны предполагают, что Devin видит этот репозиторий; линкуйте
конкретные файлы — он сразу сможет опереться.

## Основы (пять тезисов — запомнить и применять всегда)

1. **Говорите *как*, а не только *что*.** Для сложных задач описывайте
   предпочитаемый подход в самом промпте.
   - ❌ «Добавь unit-тесты.»
   - ✅ «Добавь unit-тесты для `parse_tool_call`. Кейсы: пустой вход, битый JSON,
     юникод, вложенные вызовы. Мокай LLM-клиент через `respx`.»
2. **Скажите, с чего начать.** Имя репо, файла, функции, ссылка на доку.
3. **Защитный промптинг.** Представьте того же нового стажёра — где он запутается?
   Заранее снимите неоднозначность.
4. **Обеспечьте CI, тесты, типы, линтер.** Фидбек-луп превращает агента в сильного.
5. **Проверяйте.** Ответственность за финальную корректность — на вас.

## Анатомия хорошего промпта

```text
[Objective]      1–2 предложения. Какой результат?
[Context]        Ссылки на файлы / доки / прошлые PR / ADR.
[Approach]       Как вы хотите, чтобы это было сделано (опционально, но крайне желательно).
[Constraints]    Стек, стиль, perf-бюджет, безопасность, обратная совместимость.
[Acceptance]     Какие тесты должны пройти, какое поведение проверяется.
[Out of scope]   Чего Devin трогать не должен.
```

## Шаблоны

### T1 — Research & summarise into a note

```text
[Objective]
Исследовать <topic>. Результат — структурированная заметка
в knowledge/research/<slug>.md.

[Context]
- Стартовое чтение: <link1>, <link2>
- Связанный код в репо: <path>
- Связанный ADR: knowledge/adr/<file>

[Approach]
- Пробежаться по 3–5 самым релевантным первичным источникам.
- Структура заметки:
    1. TL;DR (≤ 3 предложений)
    2. Ключевые концепции
    3. Компромиссы vs наш подход
    4. Открытые вопросы

[Constraints]
- Каждая неочевидная формулировка — с URL-ссылкой.
- Кода в этом PR нет.

[Acceptance]
- Файл создан, markdownlint проходит (см. scripts/lint-md.sh, когда появится).
- Открой draft PR. Не переводи в ready-for-review.
```

### T2 — Scaffold a new module

```text
[Objective]
Создать src/<module>/ — чистый типизированный скелет под <purpose>.

[Context]
- Существующий паттерн: src/<other_module>/
- Черновик публичного API:
    class X:
        def run(self, inp: InputT) -> OutputT: ...

[Approach]
- Добавить запись в pyproject, __init__, основной класс,
  один happy-path и один error тест.
- Подключить в CI (pytest collection path, если требуется).

[Constraints]
- Python 3.11+, полные type-хинты, ruff clean, mypy --strict clean для модуля.
- В тестах — без сети; LLM-клиент мокается.

[Acceptance]
- `pytest src/<module>` зелёный, `ruff check .` чист, `mypy src/<module>` чист.
- PR включает короткий README.md модуля.

[Out of scope]
- Не рефакторить соседние модули.
```

### T3 — Compare two approaches (анти-«analysis paralysis»)

```text
[Objective]
Нарисовать два варианта реализации <feature> в параллельных PR, чтобы сравнить.

[Context]
- Вариант A: <фреймворк / подход / ссылка>
- Вариант B: <фреймворк / подход / ссылка>
- Критерии: латентность, читаемость, покрытие тестами, простота расширения.

[Approach]
- Два отдельных PR, в каждом — минимальный сквозной слайс.
- В каждый PR добавить `DECISION_NOTES.md` с плюсами/минусами.

[Acceptance]
- Оба PR зелёные в CI.
- В каждом — короткий бенчмарк в `bench/` с воспроизводимой командой.
```

### T4 — Co-develop a PRD

```text
[Objective]
Совместно написать PRD по <feature>. Кода нет.

[Context]
- Проблема: ...
- Пользователи: ...
- Известные ограничения: ...

[Approach]
- Прочитать релевантный код/доку и выдать черновик PRD в docs/prd/<slug>.md.
- После первого драфта — пауза, жду ревью.

[Acceptance]
- PRD покрывает: цели, non-goals, user stories, набросок API, риски, план
  выкатывания, метрики.
```

### T5 — Bug reproduction first, then fix

```text
[Objective]
Воспроизвести <bug> падающим тестом, ПОТОМ починить.

[Context]
- Репорт: <link>
- Подозрительная область: <path>

[Approach]
- Шаг 1: написать падающий тест, зафиксировать баг. Открыть draft PR только
  с тестом. Ждать апрува.
- Шаг 2: минимальный фикс. Пушить в тот же PR.

[Acceptance]
- Тест падает до фикса, проходит после.
- Никаких посторонних правок.
```

## Anti-patterns

- ❌ «Улучши код.» — нет критериев успеха.
- ❌ «Построй агента.» — слишком большой scope; декомпозируйте.
- ❌ Стек-трейс без контекста репо и шагов воспроизведения.
- ❌ Дать агенту тихо выбрать между двумя крупными подходами — либо решайте сами,
  либо попросите оба (T3).
- ❌ Мержить, не читая диф, даже маленький.
