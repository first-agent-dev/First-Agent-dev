CLASS: REPAIR
INVARIANT: Affects: LLM request assembly must preserve provider tool-call/result pairing; provider usage must be audit-visible per turn and summarized per session; tool-result payloads must cross into model context only through a single budgeted projection boundary while the audit log retains full result payloads.

DEGREE-OF-FREEDOM CLOSED: The previous loop sent only `result.summary` directly into provider messages, did not record prompt-cache usage counters, did not write run-level cache summaries, did not test cache SLOs, did not enforce the tool-call/result pairing invariant before provider requests, and did not have per-tool context budgets.

DETERMINISTIC MECHANISM: `drive_session()` now records `usage` rows and a final `session_summary`, asserts `_assert_tool_pairing_invariant(messages)` under `__debug__` before each provider request, and appends tool messages via `project_for_model(spec, result, artifact_store)` instead of directly using `result.summary`.

## Summary

This PR lands the first loop-optimization foundation slices from the tiered plan:

1. Tier 0 instrumentation: cache-aware usage fields, per-turn `usage` rows, session-level cache summary, and a recorded-transcript cache SLO test.
2. Pairing invariant: a debug assertion before every provider request catches orphaned tool calls/results locally before the provider returns a 400.
3. Canonical tool serialization: tools are sorted by name and canonicalized through stable JSON.
4. Tool projection chokepoint: `ToolSpec` carries context budget/elision knobs; `project_for_model()` is the single boundary from full `ToolResult` to model-visible text; elided payloads are written to a deterministic artifact store.
5. Audit fidelity: `SessionState.record_tool_result()` now records full successful `ToolResult.result` payloads in `events.jsonl` while provider messages receive only the projected string.

## Code Changes

### Provider usage shape

- `src/fa/providers/base.py`
  - Adds `ResponseInfo.cache_read_input_tokens` and `ResponseInfo.cache_creation_input_tokens`, defaulting to zero.

- `src/fa/providers/anthropic.py`
  - Extracts Anthropic `usage.cache_read_input_tokens` and `usage.cache_creation_input_tokens`.

- `src/fa/providers/openai_compat.py`
  - Extracts OpenAI-compatible cached-token reads from `usage.prompt_tokens_details.cached_tokens`.
  - Also accepts provider-normalized `cache_read_input_tokens` / `cache_creation_input_tokens` if present.

### Tier 0 usage logging and session summary

- `src/fa/inner_loop/coder_loop.py`
  - Adds `_usage_event_content()`.
  - Adds `_session_summary_content()`.
  - Writes one `kind="usage"` row for every successful provider response.
  - Writes one `kind="session_summary"` row on every controlled terminal path.
  - Computes:
    - `input_tokens`
    - `cache_read_input_tokens`
    - `cache_creation_input_tokens`
    - `uncached_input_tokens`
    - `output_tokens`
    - `cache_hit_ratio`

- `src/fa/inner_loop/state.py`
  - Documents the new `usage` and `session_summary` event kinds.

### Tool pairing invariant

- `src/fa/inner_loop/coder_loop.py`
  - Adds `_assert_tool_pairing_invariant()`.
  - Runs it in `__debug__` immediately before each `provider_chain.request()`.
  - Supports both the repo's current OpenAI-shaped history (`assistant.tool_calls` + `role="tool"`) and Anthropic-style content blocks (`tool_use` / `tool_result`) for future compatibility.
  - Canonicalizes malformed/missing tool-call ids in provider history through `_tool_calls_for_message()` so fallback ids generated by `_build_tool_calls()` also appear in the assistant message envelope. This avoids creating an orphaned tool result on the follow-up request.

### Canonical tool serialization

- `src/fa/inner_loop/prompt.py`
  - `render_tool_specs()` now sorts `ToolSpec`s by name.
  - Each tool payload is round-tripped through `json.dumps(..., sort_keys=True, separators=(",", ":"))` + `json.loads(...)` for byte-stable key ordering.

### Tool projection chokepoint

- `src/fa/inner_loop/registry.py`
  - Adds `ToolElider` alias.
  - Adds `ToolSpec.max_context_bytes: int = 4096`.
  - Adds `ToolSpec.elide: ToolElider | None = None`.
  - Rejects negative context budgets.

- `src/fa/inner_loop/artifacts.py`
  - Adds deterministic content-addressed artifact store under the run directory.
  - Artifact ids are `tool-result-<sha16>`.

- `src/fa/inner_loop/projection.py`
  - Adds `render_tool_payload()`.
  - Adds default head/tail elision.
  - Adds `project_for_model(spec, result, artifact_store)`.
  - Error results project as `ERROR[code]: <message[:500]>`.
  - Under-budget payloads include `summary + rendered payload`.
  - Over-budget payloads are elided and full payload is written to artifact storage.
  - `max_context_bytes=0` means summary plus artifact pointer only.

- `src/fa/inner_loop/coder_loop.py`
  - Replaces direct `result.summary` provider-message append with `project_for_model(...)`.
  - Looks up the emitting tool's `ToolSpec` for projection; malformed/unregistered tool calls use a safe fallback projection spec.

### Audit fidelity

- `src/fa/inner_loop/state.py`
  - Adds `_json_safe()`.
  - `record_tool_result()` now records full successful `result.result` payloads in `events.jsonl`, not just summaries/errors/artifacts.

## Review Fixes Applied During Final Pass

- Canonicalized assistant tool-call ids in message history after `_build_tool_calls()` fallback id generation. Without this, a provider response missing `tool_call.id` would produce a fallback `ToolCall.call_id` for the tool result but leave the assistant message with a missing id, causing the next pairing invariant check to fail.
- Preserved summary-writing on all controlled terminal paths through the local `finish()` helper.
- Kept usage logging tied to successful provider responses only; failed provider-chain attempts continue to be represented by `provider_attempt` rows.
- Kept compaction and assembly middleware out of this PR to avoid cache-hostile changes before metrics prove the need.
